# UX Audit: VehicleService Homepage

**Date**: 2026-03-03  
**Scenario**: "First-time visitor lands on homepage and tries to compare services, understand membership value, and book a transfer"  
**Browser**: Fallback audit via live HTML inspection (`curl`) + source review (Chrome MCP/Playwright unavailable in this environment)  
**Viewport**: Desktop markup reviewed; mobile behavior reviewed via component logic

## Summary

The homepage communicates premium positioning well and surfaces booking early, but several navigation paths lead to dead ends. Two visible CTA paths currently return 404, and shared hash navigation links fail on non-home pages. The booking flow is visually strong but lacks explanatory feedback in key incomplete/error states.

## Findings

### Critical (blocks user task)

- **Homepage CTA points to missing route (`/how-it-works`)**
  - *Where*: Join Community section
  - *Evidence*: [join-community.tsx](/Volumes/shorya/apps/vehicleservice/components/home/join-community.tsx:75)
  - *Observed*: `GET /how-it-works` returns `404`.
  - *Impact*: User intent to learn before signup fails immediately.
  - *Fix*: Add `app/how-it-works/page.tsx` or change CTA to an existing explainer route.

- **Footer "Airport Transfers / City Transfers" route to missing `/search` page**
  - *Where*: Footer service links
  - *Evidence*: [footer.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/footer.tsx:20), [footer.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/footer.tsx:21)
  - *Observed*: `GET /search` returns `404`.
  - *Impact*: Users cannot continue from a core service discovery path.
  - *Fix*: Add `app/search/page.tsx` or update links to a valid destination (`/search/results` with defaults, or a dedicated search landing route).

### High (causes confusion or significant friction)

- **Global nav uses in-page hash links that fail off-homepage**
  - *Where*: Shared public header/mobile menu used in public layouts
  - *Evidence*: [public-header.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/public-header.tsx:132), [mobile-menu/index.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/mobile-menu/index.tsx:136), [public-layout.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/public-layout.tsx:45), [contact/layout.tsx](/Volumes/shorya/apps/vehicleservice/app/contact/layout.tsx:6), [blog/layout.tsx](/Volumes/shorya/apps/vehicleservice/app/blog/layout.tsx:6)
  - *Observed*: On `/contact` and `/blog`, links to `#services`, `#fleet`, `#faq` exist, but those IDs are absent on the page.
  - *Impact*: Menu appears functional but does nothing for key destinations.
  - *Fix*: Resolve nav targets by pathname (`/#services` on non-home pages, `#services` on home), or route to dedicated pages.

- **Search submit is disabled without guidance unless both locations are selected from autocomplete**
  - *Where*: Hero booking form
  - *Evidence*: [search-form.tsx](/Volumes/shorya/apps/vehicleservice/components/home/hero/search-form.tsx:148), [search-form.tsx](/Volumes/shorya/apps/vehicleservice/components/home/hero/search-form.tsx:45)
  - *Observed*: Typing values without selecting suggestion keeps button disabled; no inline helper/error message.
  - *Impact*: Users can interpret disabled CTA as a broken form.
  - *Fix*: Show explicit helper text ("Select a location from suggestions") and inline validation state for both fields.

### Medium (suboptimal but workable)

- **Multiple placeholder links (`#`) create dead-end interactions**
  - *Where*: Footer links and social icons
  - *Evidence*: [footer.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/footer.tsx:22), [footer.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/footer.tsx:29), [footer.tsx](/Volumes/shorya/apps/vehicleservice/components/layout/footer.tsx:38)
  - *Impact*: Trust erosion in a premium brand experience.
  - *Fix*: Hide non-ready links behind a feature flag or route to lightweight "Coming soon" pages with capture CTA.

- **No "no results" state in location autocomplete**
  - *Where*: Hero location autocomplete
  - *Evidence*: [location-autocomplete.tsx](/Volumes/shorya/apps/vehicleservice/components/home/hero/location-autocomplete.tsx:79), [location-autocomplete.tsx](/Volumes/shorya/apps/vehicleservice/components/home/hero/location-autocomplete.tsx:173)
  - *Observed*: Suggestion list only renders when results exist; empty search gives no feedback.
  - *Impact*: Users do not know whether input is invalid, still loading, or unsupported.
  - *Fix*: Render an explicit empty state ("No locations found. Try city or airport name.").

- **Date defaults use UTC date string**
  - *Where*: Homepage date seed
  - *Evidence*: [page.tsx](/Volumes/shorya/apps/vehicleservice/app/page.tsx:28)
  - *Impact*: Around timezone boundaries, local "today" can mismatch the prefilled/min date.
  - *Fix*: Derive date in user-local timezone on client, or compute local date server-side from intended locale.

## What Works Well

- Hero section clearly communicates product category and value proposition.
- Booking flow is placed prominently above fold with strong visual hierarchy.
- Autocomplete supports keyboard navigation and `aria` metadata for listbox patterns.
- Public layout consistency (header/footer/currency) provides predictable structure across pages.

## Recommendations (priority order)

1. Fix broken routes linked from homepage (`/how-it-works`, `/search`) before additional UX polish.  
2. Make global navigation route-aware so section links always land correctly from non-home pages.  
3. Add booking-form guidance and autocomplete empty/error messaging to reduce first-time drop-off.
